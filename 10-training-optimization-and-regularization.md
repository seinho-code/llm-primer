# 10 训练、优化与正则化：模型为什么能学会东西

很多人刚接触 AI 时，会把注意力都放在模型架构名字上，比如 Transformer、MoE、Diffusion。但真正做模型训练的人很快会发现：很多成败并不来自“架构换没换”，而来自训练配方是否合理。

这章讲的就是那套经常被低估、但实际上非常关键的训练逻辑：

- loss 怎么定义
- optimizer 怎么选
- learning rate 怎么调
- batch size 有什么影响
- 正则化为什么重要
- 训练为什么会不稳定

## 1. 训练到底在优化什么

训练的核心目标可以写成：

`min_theta E_(x,y) [L(f(x; theta), y)]`

这句话翻成白话就是：

在所有训练样本上，找到一组参数 `theta`，让模型输出和真实目标之间的平均误差尽量小。

这里的关键对象有三个：

- `f(x; theta)`：模型
- `L(...)`：损失函数
- `theta`：参数

如果你把训练看成一个系统，那么：

- 数据负责告诉模型“什么是对的”
- 损失函数负责定义“错了多少”
- 优化器负责决定“怎么改参数”

## 2. 数据集、epoch、step、batch 是什么关系

### 2.1 Dataset

数据集就是全部训练样本。

### 2.2 Batch

一次不会把所有数据都送进模型，而是拆成很多小批次，这就是 batch。

### 2.3 Step

每处理一个 batch，并更新一次参数，通常算一个 training step。

### 2.4 Epoch

当整个数据集被完整遍历一遍，就算一个 epoch。

一个很重要的工程事实是：很多论文里真正可比的不是“训练了几个 epoch”，而是：

- 总 token 数
- 总 step 数
- 总训练 FLOPs

## 3. Loss 函数：模型错得有多离谱

常见损失函数有：

- `cross entropy`：分类和语言模型最常见
- `MSE`：回归问题常见
- `pairwise / preference loss`：排序、偏好学习常见

语言模型里最常见的是 next-token cross entropy。它的目标非常直接：

让正确下一个 token 的概率尽可能高。

但不要低估 loss 的设计，它往往在定义：

- 你希望模型学会什么
- 什么错误最不该犯
- 训练信号到底怎么传回去

## 4. Batch size 为什么会影响训练效果

Batch size 不只是吞吐参数，它还会影响优化行为。

### 4.1 小 batch

优点：

- 梯度更有噪声，有时反而更利于泛化
- 显存要求更低

缺点：

- 训练不够稳定
- 吞吐较低

### 4.2 大 batch

优点：

- 硬件利用率高
- 梯度估计更稳定

缺点：

- 显存要求高
- 需要重新调 learning rate
- 过大时可能损害泛化

因此 batch size 不是越大越好，而是要和：

- 学习率
- 优化器
- 数据规模
- 硬件资源

一起看。

## 5. 学习率：训练里最敏感的超参数之一

学习率控制每一步更新走多远。

### 5.1 学习率太大

- loss 震荡
- 梯度爆炸
- 训练直接发散

### 5.2 学习率太小

- 收敛很慢
- 容易卡在次优区域
- 训练预算被浪费

### 5.3 为什么要 warmup

很多大模型训练一开始会用 warmup，也就是学习率先从很小逐渐升高。

原因是：

- 训练初期参数和激活分布还不稳定
- 一上来就大步更新容易炸

所以 warmup 本质上是一种“起步稳一点”的策略。

## 6. Optimizer：参数更新不只是减梯度

### 6.1 SGD

最基础的优化器，简单直接。

### 6.2 Momentum

给更新加上惯性，减少左右摇摆。

### 6.3 Adam

会结合一阶和二阶统计量估计，为不同参数自适应调节步长。

### 6.4 AdamW

在现代 Transformer 训练中非常常见，因为它把 weight decay 和梯度更新更清晰地分开。

你不必一开始背会公式，但要知道：

- 不同优化器会改变训练稳定性
- 它们对学习率和 batch size 的敏感度也不同

## 7. Weight Decay 和正则化在防什么

训练集上表现很好，不代表模型真学会了。它可能只是记住了训练数据。

正则化的目标就是缓解过拟合。

常见方式：

- weight decay
- dropout
- 数据增强
- early stopping
- label smoothing

### 7.1 Weight decay

它会抑制参数无限变大，帮助模型保持更平滑的解。

### 7.2 Dropout

训练时随机屏蔽部分神经元，让模型不要过度依赖某些局部路径。

在很多现代大模型里，dropout 比早期小模型更少见，但正则化思想并没有消失，而是体现在更多训练设计中。

## 8. 梯度裁剪为什么重要

有时训练会出现梯度突然非常大，导致参数一步被更新坏掉。这时常会用 gradient clipping。

它的本质是：

- 如果梯度过大，就把它缩放回安全范围

这是一种很实用的训练稳定器，尤其在大模型、RNN、强化学习或混合训练场景中常见。

## 9. 归一化、残差和初始化：稳定训练的三件套

如果你问为什么现代深度模型能堆这么深，答案往往不是单个大理论，而是一整套稳定训练技巧一起起作用：

- 好的参数初始化
- 合适的归一化
- 残差连接

这些内容在单看某篇论文时容易被当成实现细节，但在工程上它们常常决定训练能不能跑起来。

## 10. 过拟合与欠拟合怎么判断

### 10.1 欠拟合

- 训练集和验证集都差
- 模型容量不够，或训练不充分

### 10.2 过拟合

- 训练集很好，验证集变差
- 模型记住了样本噪声而不是规律

一个成熟训练流程一定要持续看：

- train loss
- validation loss
- 目标任务指标

而不是只盯训练损失。

## 11. 数据配比为什么比想象中重要

大模型常用混合语料，例如：

- 通用网页文本
- 高质量文档
- 代码
- 数学数据
- 对话数据

不同数据比例会直接影响模型偏好：

- 代码多，编程能力更强
- 数学多，推理题更稳
- 对话多，聊天风格更顺

所以训练不是“数据越多越好”，而是“哪些数据按什么比例混合”。

## 12. 分布式训练：为什么单卡很快不够了

当模型和数据足够大时，单卡训练不现实，需要分布式。

常见并行方式：

- `data parallel`：不同卡看不同 batch
- `tensor parallel`：同一层的矩阵拆到多卡
- `pipeline parallel`：不同层放在不同卡

你不一定一开始就要自己写分布式框架，但要知道：

- 大模型训练不是“一个 Python 脚本 + 一张卡”的问题
- 通信开销和显存布局是核心约束

## 13. 训练为什么经常“不按论文来”

真实训练环境里，经常会出现：

- 数据脏
- 标签不一致
- tokenization 有坑
- 学习率 schedule 不合适
- mixed precision 数值不稳定
- checkpoint 恢复后曲线异常

因此训练工作很大一部分其实是 debugging。

## 14. 一个常见的训练问题排查框架

如果模型训练效果不对，可以先按这个顺序查：

1. 数据有没有问题
2. 标签是否对齐
3. tokenizer 是否一致
4. loss 是否正确 mask
5. 学习率是否过大或过小
6. batch size 是否改了但超参没调
7. 训练/验证指标是否被正确记录

很多“神秘问题”最后都不是模型理论问题，而是训练管线细节问题。

## 15. Scaling law 对工程的启发

Scaling law 的现实意义不只是“更大更强”，而是告诉你：

- 模型规模
- 数据量
- 训练算力

三者要平衡。

如果你只有很少数据，却训练超大模型，可能不划算；如果你数据极多、模型极小，也可能吃不满数据价值。

## 16. 训练和推理是两套不同思维

训练阶段关注：

- 收敛
- 泛化
- loss 降不降
- 训练稳定性

推理阶段关注：

- 延迟
- 吞吐
- 显存
- 成本

很多开发者只看模型结构，不区分这两套目标，容易做出错误判断。

## 17. 一个从业者应该具备的训练直觉

你不一定要天天训大模型，但最好能形成这几个判断习惯：

- 一个结果变差，是数据问题、目标问题还是优化问题
- 一个超参改动，影响的是稳定性、收敛速度还是泛化能力
- 一个训练“更大”的方案，到底是在烧算力还是在换真实收益

## 18. 小结

训练不是把数据扔进模型里等奇迹发生，而是一套由目标函数、优化器、学习率、batch、正则化、数据配比和系统资源共同构成的动态过程。很多时候，真正的工程能力就在于：你能不能判断问题出在哪一层，并用最小代价把训练拉回正轨。
